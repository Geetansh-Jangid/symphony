# Code Format Reference (Internal Only - Do Not Commit)

## Core Philosophy

Write code like you're explaining it to a competent friend. Clear, direct, no fluff.

## Research First Rule

**ALWAYS verify before using:**
- Package names: Check https://aur.archlinux.org/packages/PKGNAME (2026 sources)
- Commands: Check official docs/man pages for current syntax
- Deprecated tools: Verify still maintained before including

Use websearch/subagents for verification - don't assume.

## Shell Script Guidelines

### 1. Error Handling
- Use `|| { err "message"; return 1; }` for one-liners
- Use full if-statements for multi-step error handling
- Never use `exit` inside a function - use `return`
- Traps set inside functions must be cleared: `trap - EXIT`

### 2. Navigation
- Use `cd` only in subshells: `(cd dir && cmd)`
- Or use `pushd/popd` if you must
- Prefer running commands with full paths instead of cd

### 3. Functions
- Keep them small (< 20 lines if possible)
- Single purpose
- Return 0 on success, 1 on failure
- No side effects (don't change global state unexpectedly)

### 4. Variables
- Always quote: `"$var"` not `$var`
- Use local in functions: `local var=value`
- Use arrays for lists, not strings

### 5. Comments
- Only explain WHY, not WHAT
- If you need a comment to explain what code does, rewrite the code
- Header comments are fine for file purpose

### 6. Style
- Use tabs for indentation (consistent with existing codebase)
- One blank line between logical sections
- Group related commands
- Prefer `[[ ]]` over `[ ]` for bash

### 7. External Commands
- Check if tools exist before using them
- Provide fallbacks for optional tools
- Never silence errors with `2>/dev/null` unless you handle them

### 8. Temp Files
- Always use `mktemp`
- Always cleanup (trap EXIT)
- Reset traps after use in functions

## Workflow

1. **Research**: Use websearch/subagents to verify current info
2. **Implement**: Main agent writes code based on verified info
3. **Output**: Main agent provides summary only

## Token-Saving Research Notes

- Use omarchy pattern: `cmd || { err "msg"; return 1; }`
- Package arrays grouped by purpose with comments
- Helper functions in utils.sh, not inline
- Use existing utilities (pkg_installed, ok, err, etc.)
- Verify AUR packages at https://aur.archlinux.org/packages/NAME

## What to Avoid

- Over-abstracting (don't make a function for 2 lines used once)
- Unnecessary comments (`# increment counter` - we can see that)
- Echo debugging left in production
- Long pipelines (hard to debug, break into steps)
- Silent failures (always show errors to user)
